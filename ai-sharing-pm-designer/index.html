AI 時代角色變化

執行變快了，但判斷力沒變

  • 以前寫一份 PRD 要幾天，現在一個下午就能從想法變成 prototype
  • 「該問什麼問題」「該解決什麼問題」— 這些判斷力沒被 AI 取代
  • 反而因為執行變快了，判斷力變得更值錢
  • 工程師面對一樣的處境：AI 寫 code 越來越快，但「該寫什麼」不會被取代

AI 是加速器，
不是替代者。

PM 和設計師的價值
在「定義」而非「執行」

— Claude 官方 blog "Product Management on the AI Exponential"

METHODOLOGY

什麼是 SDD？

Spec-Driven Development — 規格驅動開發

傳統開發

20% 需求討論

80% 寫 code + 改 bug + 再改 bug

SDD 開發

80% 準備（規格 + 計畫 + 拆解）

20% AI 寫 code

六步流程

1
Constitution（定規矩）
2
Specify（寫規格 / PRD）
3
Clarify（抓漏）
4
Plan（技術藍圖）
5
Tasks（拆任務）
6
Implement（AI 照清單打勾）

前 5 步 PM & Designer 都能參與，
只有最後一步需要寫 code

STEP 3

從 Design 到精準開發

痛點：設計稿和實作總是有落差

傳統流程的問題

  • 設計師：「按照設計稿」
  • 工程師：「我覺得差不多」
  • 結果：顏色偏了、間距不對、狀態漏了

Prospec 的解法：節點式設計映射

每個設計節點都有對應的任務 → AI 照著做，不會猜測

// UI 設計節點 → 任務標記 Button (Primary CTA) → REQ-UI-001 ├─ default 狀態 ├─ 實作基礎按鈕 ├─ hover 狀態 ├─ 加上 hover 效果 ├─ loading 狀態 ├─ 加上載入動畫 └─ disabled 狀態 └─ 加上禁用樣式 Modal (Gift Form) → REQ-UI-002 ├─ 開啟動畫 ├─ overlay + z-index ├─ 表單驗證 ├─ email 格式 + 必填 └─ 響應式 └─ mobile 全寬 / desktop 600px

PRECISION

Prospec 三層精準機制

1

設計規格

design-spec.md

  • 設計令牌：顏色、字體、間距的精確值
  • 元件結構：每個元件的 props 和 states
2

互動規格

interaction-spec.md

  • 頁面流程：A → B → C 的狀態轉換
  • 手勢操作：點擊、滑動、長按
3

Delta Spec

變更追蹤

  • ADDED：新增了什麼設計
  • MODIFIED：改了什麼（Before / After）
  • REMOVED：移除了什麼（附原因）
設計令牌
（精確值）
→
Delta Spec
（追蹤變更）
→
任務清單
（打勾）
→
AI 精準實作
（不猜測）
