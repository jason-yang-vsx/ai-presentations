## User Story

As a 老師
I want to 課後查看出席報告
So that 掌握學生出席狀況

## Acceptance Criteria

✓ 課堂結束後自動生成出席清單
✓ 顯示出席/缺席/遲到狀態
✓ 缺席學生自動寄 email 通知家長
✓ 老師可手動調整出席狀態

Skill 的三層架構 全員適用

Skill Architecture

Skill 的三層架構 — Progressive Disclosure 全員適用

三層載入機制：

Level 1: Metadata
永遠在 context 中 — name + description (~100 words)

Level 2: SKILL.md Body
Skill 觸發時載入 — 完整指令 (~500 lines)

Level 3: Bundled Resources
按需載入 — scripts/ references/ assets/ (無限)

my-skill/
├── SKILL.md          ← 主要指令（必需）
├── references/    ← 詳細文件（按需載入）
├── examples/      ← 範例輸出
└── scripts/       ← 可執行腳本

SKILL.md 結構：

---
name: explain-code
description: 用圖表和類比解釋程式碼
disable-model-invocation: true
allowed-tools: Read Grep Glob
model: sonnet
context: fork
---

# 指令內容
1. 先用類比解釋
2. 畫 ASCII 圖
3. 逐步走過程式碼
4. 提醒常見陷阱

Frontmatter 關鍵欄位：

  • description — Claude 決定何時觸發的依據（最重要！）
  • disable-model-invocation: true — 只有你能觸發
  • allowed-tools — 限制可用工具（唯讀：Read Grep Glob）
  • context: fork — 在獨立 subagent 中執行
  • model — 指定模型（sonnet 省成本）

Skill Creator — Eval 驅動開發 全員適用

Skill Creator

Skill Creator — 用 Eval 驅動的 Skill 開發 全員適用

四個模式：

1. Create

捕捉意圖 → 訪談
→ 寫 SKILL.md
→ 建測試案例

2. Eval

四個 sub-agent 平行：
Executor・Grader
Comparator・Analyzer

3. Improve

根據回饋迭代：
不要 overfit
保持精簡・解釋 why

4. Benchmark

生成 HTML 報告
比較 with-skill
vs baseline

Description 是觸發的關鍵：

Bad: "How to build a dashboard"

Good: "How to build a dashboard to display internal data. Use when user mentions dashboards, data visualization, internal metrics — even if they don't explicitly ask for a 'dashboard.'"

Eval Pipeline 流程：

生成測試 → 執行（with-skill + baseline）
  → 評分 → 聚合 → 分析 → 開 Viewer
  → 讀回饋 → 改進 → 重跑

迭代原則：
「從回饋中歸納，不要 overfit 到測試案例」
「移除沒用的部分 — 看 transcript 找浪費」
「解釋 why，不要用 ALWAYS/NEVER」

Context Management

Context Window 是最重要的資源 工程師

context 越滿，AI 越笨 — 所有最佳實踐的出發點

/compact

80% 就要清理
手動壓縮歷史
可指定保留重點

/clear

任務切換時用
完全重置 context
乾淨開始新任務

/btw

快速提問
不進對話歷史
零 context 成本

Subagent

調查工作丟出去
不污染主 context
回報摘要就好

常見失敗：一個 session 做很多不相關的事（「廚房水槽 session」）
解法：一個任務一個 session，做完 /clear 再開始下一個

Prompt Tips

好 Prompt vs 壞 Prompt 全員適用

Bad — 模糊、沒驗證標準

「加測試」
「修登入 bug」
「讓 dashboard 好看一點」

Good — 具體、可驗證、有範圍

「幫 foo.py 寫測試，cover 使用者
登出的 edge case，不要用 mock」
「session timeout 後登入失敗
檢查 src/auth，先寫失敗測試再修」
「[貼截圖] 實作這個設計
完成後截圖比對，列出差異」

具體

指定檔案、限制、範例

可驗證

測試、截圖、預期輸出

有範圍

不要讓 AI 無限探索

Anti-patterns

五個常見失敗模式 全員適用

失敗模式症狀解法
廚房水槽 session 一個 session 做很多不相關的事 /clear 切換任務
反覆修正 同一問題修了兩次還不對 /clear 重寫更好的 prompt
CLAUDE.md 過度膨脹 Claude 忽略一半的指令 狠心刪減，改用 hooks 或 skills
信任但不驗證 看起來對但 edge case 沒處理 一定要有驗證手段
無限制探索 叫 AI 調查但沒限範圍 限縮範圍或用 subagent

核心原則：及早修正 — Esc 停止、Esc Esc 回退到 checkpoint
